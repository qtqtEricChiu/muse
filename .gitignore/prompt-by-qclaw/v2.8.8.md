## 问题描述

### 问题1: 空翻译行错误处理

**用户反馈**:
> "2.8.6对于双语歌词得到了修复，但是出现了一个问题，就是有时候如果一个原文歌词翻译空置了，这个时候本应该直接展示单行歌词，但是代码选择重复放一段下一段歌词（作为所谓的翻译歌词），这种选择是不正确的，应当直接展示该单行歌词而不做处理"

**复现条件**:
1. LRC 文件中有时间戳组的**第一行为空**（翻译行空置）
2. 例如：
   ```
   [00:12.50] Hello world
   [00:15.30] 
   [00:15.30] Goodbye world
   ```
   这里 `[00:15.30]` 的第一行（翻译）是空的

**预期行为**:
- 如果翻译为空，应该将该歌词作为**单行歌词**显示（不显示翻译行）

**实际行为**:
- 代码将下一句原文（`Goodbye world`）错误地当作翻译
- 显示：`Hello world` + `Goodbye world`（作为翻译）

---

### 问题2: 翻译行字号与原文太接近

**用户反馈**:
> "主界面歌词框的双语歌词中，第二行歌词的字号应该小于原文歌词，而不是一样大"

**复现条件**:
1. 播放双语歌词歌曲
2. 观察激活行的原文和翻译的字号差异

**预期行为**:
- 翻译行字号应该**明显小于**原文（例如 80% 或 75%）

**实际行为**:
- 激活状态：翻译行 `font-size: 0.95em`（仅小5%）
- 非激活状态：翻译行 `font-size: 0.8em`（小20%，但激活时又变大了）

## 根因分析

### 根因1: `parseLyricText()` 未检查翻译是否为空（最高优先级）

**位置**: `app.js` 第1340-1355行

**问题代码**:
```javascript
// Step2: 按时间戳分组（保持同时间戳内顺序）
const groups = [];
for (const entry of entries) {
    const last = groups[groups.length - 1];
    if (last && Math.abs(last.time - entry.time) < 0.001) {
        last.entries.push(entry.text);
    } else {
        groups.push({ time: entry.time, entries: [entry.text] });
    }
}

// Step3: 链式构建歌词 — 翻译顺延到上一句
const result = [];
let pendingOriginal = null; // 等待翻译赋值的上一句原文

for (const group of groups) {
    // 过滤空行
    const texts = group.entries.filter(t => t && t.trim());
    if (texts.length === 0) continue;

    // 如果上一句有未配对的原文，本组第一行 = 它的翻译
    if (pendingOriginal && result.length > 0) {
        result[result.length - 1].translation = texts[0];  // ← 问题在这里
        result[result.length - 1].isBilingual = true;
        result[result.length - 1].text =
            result[result.length - 1].original + '\n' + texts[0];
    }

    // 本组最后一行 = 新的原文
    const original = texts[texts.length - 1];
    result.push({
        time: group.time,
        text: original,
        original: original,
        translation: null,
        isBilingual: false
    });
    pendingOriginal = original;
}
```

**问题分析**:
1. `group.entries.filter(t => t && t.trim())` 会**过滤掉空字符串**
2. 如果 LRC 格式为：
   ```
   [00:12.50] Hello world
   [00:15.30] 
   [00:15.30] Goodbye world
   ```
   解析后：
   - Group 1: `entries = ["Hello world"]`
   - Group 2: `entries = ["", "Goodbye world"]`
   - 过滤后：`texts = ["Goodbye world"]`（空字符串被过滤掉了）
3. 代码执行 `result[result.length - 1].translation = texts[0]`
   - 此时 `texts[0] = "Goodbye world"`（下一句原文）
   - 错误地将下一句原文当作翻译！

**正确逻辑**:
- 在过滤之前，检查 `group.entries[0]` 是否为空
- 如果为空，表示翻译空置，应该将该歌词作为**单行歌词**处理
- 不应该将 `texts[0]`（下一句原文）错误地当作翻译

### 根因2: CSS 翻译行字号设置不合理

**位置**: `css/base-layout.css` 第233-245行

**问题代码**:
```css
/* 激活状态：原文高亮，翻译跟随提升 */
.lrc-line.active .lrc-translation {
    opacity: 0.9;
    color: var(--text-main);
    font-size: 0.95em;  /* ← 问题：仅比原文小5%，不够明显 */
    font-weight: 500;
}

/* 非激活状态：翻译淡化 */
.lrc-line:not(.active) .lrc-translation {
    opacity: 0.45;
    font-size: 0.8em;  /* ← 这个还好，但激活时又变大了 */
}
```

**问题分析**:
1. 激活状态：`.lrc-line.active` 的 `font-size` 是 `calc(var(--lrc-font-size) * 1.44)`
2. 翻译行 `font-size: 0.95em` = `1.44 * 0.95 = 1.368` 倍基础字号
3. 原文是 `1.44` 倍基础字号
4. 差异只有 `1.44 - 1.368 = 0.072` 倍基础字号（约7.2%），**视觉上不明显**

**正确逻辑**:
- 翻译行字号应该**明显小于**原文
- 建议：`0.8em` 或 `0.75em`（比原文小20%-25%）

## 修复方案

### 修复1: 修改 `parseLyricText()` 空翻译检测逻辑

**文件**: `js/app.js`  
**函数**: `parseLyricText()`  
**行数**: 第1307-1370行  

**原代码**:
```javascript
for (const group of groups) {
    // 过滤空行
    const texts = group.entries.filter(t => t && t.trim());
    if (texts.length === 0) continue;

    // 如果上一句有未配对的原文，本组第一行 = 它的翻译
    if (pendingOriginal && result.length > 0) {
        result[result.length - 1].translation = texts[0];
        result[result.length - 1].isBilingual = true;
        result[result.length - 1].text =
            result[result.length - 1].original + '\n' + texts[0];
    }

    // 本组最后一行 = 新的原文
    const original = texts[texts.length - 1];
    result.push({
        time: group.time,
        text: original,
        original: original,
        translation: null,
        isBilingual: false
    });
    pendingOriginal = original;
}
```

**修复后代码**:
```javascript
for (const group of groups) {
    // 🩹 v2.8.8: 不立即过滤，先检查第一行是否为空（翻译行）
    const hasTranslation = group.entries.length > 1 && group.entries[0] && group.entries[0].trim();
    
    // 过滤空行（用于提取原文）
    const texts = group.entries.filter(t => t && t.trim());
    if (texts.length === 0) continue;

    // 如果上一句有未配对的原文，检查本组第一行是否为有效翻译
    if (pendingOriginal && result.length > 0 && hasTranslation) {
        // 🩹 v2.8.8: 只有当翻译非空时才赋值
        result[result.length - 1].translation = texts[0];
        result[result.length - 1].isBilingual = true;
        result[result.length - 1].text =
            result[result.length - 1].original + '\n' + texts[0];
    }
    // 🩹 v2.8.8: 如果翻译为空（!hasTranslation），不执行任何操作
    // 上一句原文保持 isBilingual = false，显示为单行歌词

    // 本组最后一行 = 新的原文
    const original = texts[texts.length - 1];
    result.push({
        time: group.time,
        text: original,
        original: original,
        translation: null,
        isBilingual: false
    });
    pendingOriginal = original;
}
```

**关键改动**:
1. 新增 `hasTranslation` 变量，检查 `group.entries[0]` 是否为空
2. 只有当 `hasTranslation = true` 时，才赋值 `translation`
3. 如果翻译为空，上一句原文保持 `isBilingual = false`，显示为单行歌词

**边界情况处理**:
- 如果 `group.entries.length === 1`（只有原文，没有翻译），`hasTranslation = false`
- 如果 `group.entries[0]` 是空字符串或只有空格，`hasTranslation = false`
- 如果 `group.entries[0]` 是非空文本，`hasTranslation = true`

### 优化2: 支持多种空翻译格式

当前代码只检查 `group.entries[0].trim()` 是否为空。但有些 LRC 文件可能使用特殊标记表示空翻译（例如 `[00:15.30] ///` 或 `[00:15.30] ---`）。

**增强检测逻辑**:
```javascript
const isEmptyTranslation = (text) => {
    if (!text || !text.trim()) return true;
    const trimmed = text.trim();
    // 特殊标记：///  ---  ???  等
    return /^[\/\-\?]+$/.test(trimmed);
};

const hasTranslation = group.entries.length > 1 
    && group.entries[0] 
    && !isEmptyTranslation(group.entries[0]);
```

## 常见问题排查

### Q1: 修复后仍然显示错误的翻译？

**排查步骤**:
1. 打开 DevTools → Sources → 设置断点在 `parseLyricText()` 中
2. 重新加载 LRC 文件
3. 检查 `group.entries` 的内容
4. 检查 `hasTranslation` 的值是否为 `false`

**可能原因**:
- LRC 文件格式不正确（时间戳不匹配）
- `group.entries[0]` 不是空字符串，而是空格或其他不可见字符
- 需要增强 `isEmptyTranslation()` 检测逻辑

---

### Q2: 翻译行字号仍然与原文太接近？

**解决方法**:
1. 打开 DevTools → Elements → 选择 `.lrc-line.active .lrc-translation`
2. 手动修改 `font-size` 的值（例如 `0.7em`、`0.65em`）
3. 观察视觉效果
4. 确定合适的值后，更新 CSS 文件

**建议值**:
- `0.75em`：平衡（推荐）
- `0.7em`：更明显
- `0.65em`：最大差异

---

### Q3: 非激活翻译行太小，看不清？

**解决方法**:
1. 修改 `.lrc-line:not(.active) .lrc-translation` 的 `font-size`
2. 建议值：`0.85em` 或 `0.9em`

**新代码**:
```css

    opacity: 0.45;
    font-size: 0.85em;  /* ← 调大 */
```
